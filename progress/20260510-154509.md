## 背景

本次工作基于 `docs/architecture-plan.md` 的初始技术方案展开，目标是将 `my-notion-go` 从一个方案目录推进为可独立运行、可独立展示、可后续持续开发的 React + Go monorepo 工程。

`my-notion-go` 当前定位为 My-Notion 的 Go Edition：不直接迁移现有 Convex / Next.js / Expo 链路，而是用 React + Go + PostgreSQL 从零实现 Web 端核心能力，后续逐步对齐原版 My-Notion 的 AI 原生能力和移动端能力。

## 风险与注意事项

1. 不要过早微服务化。
   - 当前继续保持 modular monolith。
   - 只保留 `api` 和 `worker` 两个进程。

2. 不要过早做多人协同编辑。
   - 多人协同会引入 Yjs / CRDT / OT 复杂度。
   - 当前先做单用户编辑和自动保存。

3. 不要把 RabbitMQ、Qdrant、WebSocket 一次性全部铺开。
   - 先完成 Auth、Document、Editor、AI 的主链路。
   - 再逐步引入异步任务和实时能力。

4. 保持和原版 My-Notion 的概念对齐。
   - 文档树、AI Chat、知识库、移动端能力都可以参考原版。
   - 但实现上保持 Go Edition 的独立架构。
