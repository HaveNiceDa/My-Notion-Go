# 2026-05-15 22:26:41 BlockNote renderSpec 问题修复

## 背景

- 打开文档详情页时，BlockNote 初始化阶段持续报错 `RangeError: Invalid array passed to renderSpec`。
- 后端 content API 返回内容为合法空数组，前端 sanitize 后也能生成合法 BlockNote document，因此问题不在业务正文 JSON。

## 排查过程

- 使用 `TRAE-debugger` 建立 `blocknote-renderspec` 调试会话，通过浏览器端埋点采集初始化内容、editor document、StrictMode mount 顺序和 ProseMirror renderSpec 入参。
- 先后排除：
  - 后端返回脏数据。
  - React Query 缓存混入旧内容。
  - 单纯的 React StrictMode 双挂载。
  - 单纯的 `@blocknote/mantine` UI 适配层问题。
- 参考原始 My-Notion 可运行版本矩阵，对齐：
  - `@blocknote/*` 到 `0.49.0`。
  - `react` / `react-dom` 到 `19.1.0`。
  - `@mantine/core` / `@mantine/hooks` 到 `8.3.18`。
  - `@tiptap/*` 到原项目实际工作的 `3.22.x` 组合。

## 根因

- 深层埋点确认 ProseMirror `DOMSerializer.renderSpec` 收到的结构为 `{ dom, contentDOM }`。
- 当前 ProseMirror `renderSpec` 只按 DOMOutputSpec 数组/字符串解析该入口，遇到对象时会读取 `structure[0]`，最终抛出 `Invalid array passed to renderSpec`。
- BlockNote 的部分 block `renderHTML` 会返回 `{ dom, contentDOM }`，因此需要在前端做兼容处理。

## 修复

- 在 `DocumentEditor.tsx` 中安装一个小型兼容 shim：
  - 包裹 `DOMSerializer.renderSpec`。
  - 当入参是 `{ dom, contentDOM }` 时直接返回该 DOM 结构。
  - 其他结构继续走 ProseMirror 原始逻辑。
- 空正文时不再手动传入默认 paragraph，而是让 BlockNote 自己创建默认 initialContent。
- 恢复 `@blocknote/mantine/style.css`，并补充 `@blocknote/core/fonts/inter.css`。
- 文档编辑器暂时关闭默认 `formattingToolbar` 和 `slashMenu`，与原始 My-Notion 的编辑器用法保持接近。

## 验证

- 用户刷新并重新打开文档详情页后确认编辑器正常，不再出现 `renderSpec` 报错。
- `pnpm --filter @my-notion-go/web typecheck` 通过。
- VS Code 诊断仅剩 `blocknote` / `renderspec` 拼写提示，无类型错误。
